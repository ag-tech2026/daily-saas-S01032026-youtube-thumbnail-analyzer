<task type="auto">
  <name>Task 2: Configure Google OAuth, credit init hook, and session extension in BetterAuth</name>
  <files>src/lib/auth.ts, src/lib/auth-client.ts</files>
  <action>
1. Update `src/lib/auth.ts` to add:

   a. **Imports:** Add `import { eq } from "drizzle-orm"` and `import { user } from "./schema"`. Also check if `createAuthMiddleware` needs to be imported from `better-auth/api` for hooks.

   b. **baseURL:** Add `baseURL: process.env.BETTER_AUTH_URL || process.env.NEXT_PUBLIC_APP_URL || "http://localhost:3000"` to the betterAuth config. This is CRITICAL for OAuth callbacks to work correctly (Research Pitfall 1).

   c. **socialProviders:** Add Google OAuth config:
      ```typescript
      socialProviders: {
        google: {
          clientId: process.env.GOOGLE_CLIENT_ID as string,
          clientSecret: process.env.GOOGLE_CLIENT_SECRET as string,
        },
      },
      ```

   d. **hooks:** Add after hook for credit initialization on signup. Use `createAuthMiddleware` from `better-auth/api`:
      ```typescript
      hooks: {
        after: createAuthMiddleware(async (ctx) => {
          if (ctx.path.startsWith("/sign-up")) {
            const newSession = ctx.context.newSession;
            if (newSession) {
              await db
                .update(user)
                .set({ credits: 3 })
                .where(eq(user.id, newSession.user.id));
            }
          }
        }),
      },
      ```
      Note: The DB default(3) already handles the base case. This hook is defense-in-depth and covers OAuth signup paths where user creation might set credits differently. Per research, this also covers edge cases with the Google OAuth `/sign-up/social` path -- test that `ctx.path` matching works for OAuth signup. If BetterAuth uses a different path for social signup, the hook path check may need to be `/sign-up` OR `/callback` -- check BetterAuth docs and adjust.

   e. **session:** Add session configuration to include credits in session data. Check BetterAuth v1.4.18 docs for the correct API. Research suggests `session.customFields` or a callback approach. Try:
      ```typescript
      user: {
        additionalFields: {
          credits: {
            type: "number",
            fieldName: "credits",
            returned: true,
          },
        },
      },
      ```
      This tells BetterAuth to include the `credits` field from the user table in session responses. If this API doesn't exist, fall back to using `session.select` or a custom session callback per BetterAuth docs.

      Also add explicit session timing:
      ```typescript
      session: {
        expiresIn: 60 * 60 * 24 * 7,  // 7 days
        updateAge: 60 * 60 * 24,        // Refresh after 1 day
      },
      ```

2. Update `src/lib/auth-client.ts` to extend the client type so TypeScript knows about the `credits` field on `session.user`. The approach depends on how BetterAuth exposes custom fields. If using `additionalFields`, the type should flow through automatically. If not, add a type declaration:
   ```typescript
   // If needed for type safety
   declare module "better-auth/react" {
     interface User {
       credits: number;
     }
   }
   ```
   Test with `pnpm typecheck` -- if `session.user.credits` is already typed correctly from the `additionalFields` config, skip the manual declaration.

IMPORTANT: The session extension is the trickiest part. BetterAuth's API for extending session fields has an open question in the research. The executor should:
1. First try `user.additionalFields` approach
2. If that doesn't work, check BetterAuth docs for `session.select` or `customFields`
3. As a last resort, create a separate API endpoint `/api/user/credits` that fetches credits directly from DB
