# Roadmap: Poker Hand Review AI

## Overview

This roadmap transforms an existing Next.js boilerplate with auth, database, storage, and AI chat into a poker hand analysis SaaS. The journey progresses from foundation (database schema, Google OAuth, credit system) through core features (screenshot upload, background job processing, AI vision analysis, payments) to polish (dashboard, landing page, final testing). Each phase delivers a coherent, verifiable capability that builds toward the goal: users upload poker screenshots and get instant, beginner-friendly GTO analysis.

## Phase Details

### Phase 1: Foundation & Auth
**Goal**: Users can sign up with Google OAuth and see their credit balance
**Depends on**: Nothing (first phase, brownfield — builds on existing boilerplate)
**Requirements**: AUTH-01, AUTH-02, AUTH-03, AUTH-04, CRED-02, DASH-04
**Success Criteria** (what must be TRUE):
  1. User can sign up with email/password (existing functionality verified working)
  2. User can log in with Google OAuth and account is created automatically
  3. New users receive exactly 3 free credits on first signup
  4. User can see their current credit balance displayed in site header
  5. User session persists when browser is closed and reopened
**Plans:** 2 plans

Plans:
- [ ] 01-01-PLAN.md — Schema + auth config (credits column, Google OAuth, credit init hook, session extension)
- [ ] 01-02-PLAN.md — UI integration (Google sign-in button, credit balance display in header + dashboard)

### Phase 2: Upload & Storage Pipeline
**Goal**: Users can upload poker screenshots and system deducts credits atomically
**Depends on**: Phase 1 (requires credit balance to exist)
**Requirements**: ANLYS-01, ANLYS-02, ANLYS-03, CRED-01, CRED-03
**Success Criteria** (what must be TRUE):
  1. User can drag-and-drop a poker screenshot (PNG/JPG) onto upload area
  2. User can click file picker to select poker screenshot from filesystem
  3. System rejects files over 10MB with clear error message
  4. System rejects non-image files (PDF, DOCX, etc.) with clear error message
  5. User sees "insufficient credits" prompt when trying to upload with zero balance
  6. Screenshot appears in Vercel Blob storage after successful upload
  7. Exactly 1 credit is deducted from user balance before upload completes
  8. Analysis record is created in database with "pending" status
**Plans:** 2 plans

Plans:
- [ ] 02-01-PLAN.md — Backend: analysis table schema, storage config update, POST /api/upload endpoint with atomic credit deduction
- [ ] 02-02-PLAN.md — Frontend: upload page with react-dropzone drag-and-drop, client validation, insufficient credits dialog, dashboard link

### Phase 3: Background Job Setup
**Goal**: Upload triggers background job that processes asynchronously with retry support
**Depends on**: Phase 2 (requires upload API to trigger events)
**Requirements**: ANLYS-04
**Success Criteria** (what must be TRUE):
  1. Inngest client is initialized and connected to Inngest Cloud
  2. Test function executes successfully when triggered from upload API
  3. Failed jobs automatically retry up to 3 times with exponential backoff
  4. Inngest dashboard shows event history and job status
  5. Analysis status changes from "pending" to "processing" when job starts
**Plans:** 1 plan

Plans:
- [ ] 03-01-PLAN.md — Install Inngest, create typed client and event schemas, define process-analysis function with step.run() DB status update, serve handler at /api/inngest, wire upload API to trigger events

### Phase 4: AI Vision Analysis
**Goal**: Users receive accurate GTO analysis of their poker hand within 30 seconds
**Depends on**: Phase 3 (requires background job infrastructure)
**Requirements**: ANLYS-05, ANLYS-06, ANLYS-07, ANLYS-08, ANLYS-09, ANLYS-10, ANLYS-11, CRED-04
**Success Criteria** (what must be TRUE):
  1. OpenRouter vision model receives screenshot and returns structured analysis
  2. Analysis includes verdict for each player action (preflop, flop, turn, river) with GTO assessment
  3. Analysis explains in beginner-friendly language what GTO recommends and why
  4. Analysis includes range recommendations (which hands should be in range at key decisions)
  5. Analysis includes EV notes (expected value impact of major decisions)
  6. Analysis includes overall verdict ("GTO-Compliant" or "Needs Improvement") with summary
  7. Analysis is saved to database with "complete" status when successful
  8. User receives credit refund if analysis fails after 3 retry attempts
  9. Analysis works correctly for screenshots from PokerStars, GGPoker, and 888poker
**Plans:** 1 plan

Plans:
- [ ] 04-01-PLAN.md — Zod analysis schema, GTO coach system prompt, vision analysis step in Inngest function, DB save, credit refund on failure

### Phase 5: Payment Integration
**Goal**: Users can purchase 50-credit packs for $9 via Polar checkout
**Depends on**: Phase 4 (requires working analysis flow to validate purchase value)

**Success Criteria** (what must be TRUE):

  2. User can complete purchase with credit card via Polar
  3. Webhook receives checkout.completed event after successful payment
  4. Exactly 50 credits are added to user balance after webhook confirmation
  5. Duplicate webhook events (retries) do not add credits multiple times
  6. User sees updated credit balance immediately after purchase completes
**Plans:** 1 plan

Plans:
- [ ] 05-01-PLAN.md — Polar SDK + BetterAuth plugin (checkout, webhook handler with idempotent credit fulfillment), purchase UI (dashboard buy button, zero-credit dialog, clickable header credits, processing indicator)

### Phase 6: Frontend Dashboard
**Goal**: Users can view their analysis history and see detailed results
**Depends on**: Phase 5 (requires complete analysis flow to have history to display)
**Requirements**: DASH-01, DASH-02, DASH-03
**Success Criteria** (what must be TRUE):
  1. User can see list of all past hand analyses sorted by newest first
  2. Each history item shows date, poker site (if detected), and overall verdict
  3. User can click on any past analysis to view full detail page
  4. Detail page shows all 4 analysis sections (action, range, EV, verdict)
  5. Status badges clearly indicate pending/processing/complete/failed states
  6. Page automatically polls every 3 seconds when analysis is pending/processing
  7. Polling stops automatically when analysis reaches complete/failed status
**Plans:** 2 plans

Plans:
- [ ] 06-01-PLAN.md — API routes (list + single) and dashboard history list with status badges
- [ ] 06-02-PLAN.md — Analysis detail page with polling, full result rendering, and failed state handling
